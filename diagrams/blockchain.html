How Blockchain Hashing Creates Immutability

Each block contains the hash of its own data AND the hash of the previous block. This creates a cryptographic chain. If you change data in any block, its hash changes. The NEXT block stored the OLD hash — now they don't match. Every subsequent block becomes invalid.

Try it: Edit the data in Block 2 above and watch the cascade.
1
Each block hashes its own content. The data field (transactions, records, etc.) is run through SHA-256 to produce a fixed-length digest. Any change to the data, no matter how small, produces a completely different hash.
2
Each block stores the previous block's hash. This is the "previous hash" field. It's how the chain is formed — each block cryptographically references the one before it.
3
Changing data in Block N invalidates Block N+1. Block N's hash changes → Block N+1's "previous hash" no longer matches → Block N+1's own hash changes → Block N+2 is now invalid → cascade.
4
In a distributed network, tampering is detectable. Every node holds a copy of the chain. A tampered chain won't match the majority — the network rejects it. An attacker would need to recompute ALL subsequent blocks AND control >50% of the network simultaneously.
5
Exam takeaway: Blockchain uses hashing for integrity, not confidentiality. SHA-256 (or similar) is the hash function. The immutability comes from the chained hash structure, not encryption. This is why the SEC+ exam links blockchain to the hashing domain.
