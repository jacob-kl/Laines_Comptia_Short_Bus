Exam rule: MFA requires TWO DIFFERENT categories. Password + security question = NOT MFA (both "know"). Password + TOTP = MFA. SMS OTP = weakest "have" (SIM swap). FIDO2/passkeys = phishing-resistant, no codes to steal.
User
Tries to access app.company.com (Service Provider). Not logged in.
↓
Service Provider
Generates SAML AuthnRequest, redirects browser to Identity Provider (IdP) with the request encoded in URL.
↓
Identity Provider
Presents login page. User enters credentials + MFA. IdP authenticates the user against its directory (AD, LDAP).
↓
Identity Provider
Creates a signed SAML Assertion (XML). Contains user attributes (email, groups, role). Signs with IdP private key. POSTs to SP via user's browser.
↓
Service Provider
Validates the SAML signature using IdP's public key. If valid, creates local session. Grants access. No password ever sent to SP.
Exam tip: SAML is XML-based, enterprise SSO, IdP issues signed assertions. SP never sees or stores user passwords — it trusts the IdP's signed assertion. If the IdP is compromised, all connected SPs are at risk (this is why SSO accounts need the strongest MFA).
User
Enters password at Windows logon. Client derives an encryption key from the password. Sends AS-REQ to the KDC Authentication Service.
↓
KDC / AS
Verifies user identity. Issues TGT (Ticket Granting Ticket) encrypted with the KDC's KRBTGT key. Also issues a session key. Password NOT stored or retransmitted.
↓
User
Wants to access a file server. Sends TGS-REQ to Ticket Granting Service, presenting the TGT and requesting a service ticket.
↓
KDC / TGS
Decrypts TGT, validates it. Issues Service Ticket (ST) encrypted with the target service's key. User cannot read the ST — only the service can.
↓
File Server
Decrypts the Service Ticket with its own key. Verifies contents. Grants access. No password ever reaches the file server.
Kerberoasting: Any domain user can request a service ticket for any SPN. The ticket is encrypted with the service account's password hash. Attackers extract this and crack it offline. Fix: gMSA (auto-rotating 120-char passwords, computationally infeasible to crack).
User
Clicks "Sign in with Google" on a third-party app (Relying Party). App redirects to Google with an authorization request including requested scopes.
↓
Google (IdP)
Authenticates user. Shows consent screen: "App X wants to read your profile and email." User approves. Google returns an Authorization Code to the app.
↓
App (RP)
Exchanges Authorization Code (server-to-server) for:
• Access Token — call Google APIs on user's behalf
• ID Token (JWT) — contains identity claims (sub, email, name)
↓
App (RP)
Validates ID Token signature and claims. Creates local session. User is authenticated without the app ever seeing their Google password.
OAuth 2.0 vs OIDC: OAuth alone = authorization only (access token for API calls). OpenID Connect = OAuth + identity (adds ID token with user info). NEVER use OAuth 2.0 alone for login — use OIDC. The ID token is a JWT; verify signature, issuer, audience, and expiry.
